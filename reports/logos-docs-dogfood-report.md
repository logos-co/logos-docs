## Executive Summary

A first-time external consumer following `docs/blockchain/quickstart-guide-for-the-logos-blockchain-node.md` on a Raspberry Pi 5 will hit a blocking 404 on the very first download command: the doc tells them to `wget` the `v0.4.1` circuits archive for release `0.1.2`, but release `0.1.2` actually ships `v0.4.2` circuits. After substituting the correct version, every other documented step (extract, install circuits, `init`, run node, `curl /cryptarchia/info`, `curl /network/info`) works on aarch64; the node connected to 23–24 testnet peers and processed blocks during bootstrap as the doc describes. So the quickstart is fundamentally close to correct, but the example command publicly fails today.

The Priority 1 surface has three other recurring problems that affect every reviewed doc:

1. **Eight `README.md` files are completely empty** (0 bytes), including the top-level `docs/README.md` that any GitHub user lands on. `CONTRIBUTING.md` is also a 3-line stub with one heading and no body.
2. **Repository renames have not been propagated** to the docs. `logos-co/logos-app`, `logos-co/logos-app-poc`, `logos-messaging/nwaku`, and `logos-messaging/logos-messaging-nim-compose` all silently 301 to new names. Today these still work (GitHub redirects, git follows), but every internal anchor link, branch reference, or pinned `@sha` against the old names is fragile.
3. **Half of the journey docs explicitly self-flag as stubs** (`Status: Stub`, large `[!IMPORTANT]` warnings, `UNKNOWN` in nearly every prerequisite/configuration field). They link to GitHub issues `#167`, `#174`, `#141`, `#145` as "tracking" — useful, but it means a reader navigating from the top-level README into a journey often arrives at "this page is not ready to follow."

The blockchain quickstart claims `Linux x86_64, Raspberry Pi OS (Raspberry Pi 5)` and `glibc 2.39 or later`. We verified: glibc 2.39+ excludes default Pi OS Bookworm (glibc 2.36); only Pi OS Trixie (or another distro on glibc 2.39+) works. That's a real Priority 1 prerequisite gap — telling a reader "Pi OS" without a release qualifier will produce a glibc mismatch on the most common Pi OS image at the time of writing.

### Finding F-2: glibc 2.39+ requirement is incompatible with default Pi OS Bookworm

- **Severity**: P1
- **Classification**: DOC_GAP, MISSING_PREREQUISITE, ARM_COMPATIBILITY
- **Document**: `docs/blockchain/quickstart-guide-for-the-logos-blockchain-node.md`
- **Section**: Prerequisites (lines 21–29)
- **Flow**: Install dependencies on a Raspberry Pi
- **Expected**: A reader who follows "Raspberry Pi 5 with [Raspberry Pi OS] installed" can run the node binary.
- **Actual**: The binary requires glibc ≥2.39. Default Pi OS Bookworm (the most widely deployed Pi OS at time of writing) ships glibc 2.36. Pi OS Trixie ships glibc 2.41. The doc says "Raspberry Pi OS" without specifying a release, so a reader who downloads the current stable Pi OS image and follows the quickstart will hit a glibc mismatch when running the binary.
- **Reproduction**: `docker run --rm debian:bookworm bash -c 'ldd --version | head -1'` → `2.36`. `docker run --rm debian:trixie bash -c 'ldd --version | head -1'` → `2.41`. (Pi OS releases mirror Debian glibc.)
- **Evidence**: doc text "OS: Linux x86_64, Raspberry Pi OS (Raspberry Pi 5)" + "On Linux, dependencies: glibc version 2.39 or later".
- **Root cause hypothesis**: Authors assumed "Raspberry Pi OS" implies the current Trixie image; but Pi OS Bookworm is still common and is what `rpi-imager` defaults installs land on for many users.
- **Suggested fix**: Add an explicit "Raspberry Pi OS Trixie or later" qualifier, or list a glibc check command (`ldd --version`) right under Prerequisites with a note: "On Pi OS Bookworm, the binary will fail to load; upgrade to Pi OS Trixie or run inside a `debian:trixie` Docker container."

### Finding F-6: "1 block every 10 seconds" is misleading during bootstrap

- **Severity**: P3
- **Classification**: DOC_AMBIGUITY
- **Document**: `docs/blockchain/quickstart-guide-for-the-logos-blockchain-node.md`
- **Section**: Step 3 (line 152)
